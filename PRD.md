# Xuwu Ledger - 业务流转与结算系统 PRD

## 1. 产品概述

### 1.1 产品背景

当前业务涉及销售、客户、运营审核（旭吾内部）及外部链条方等多方角色，核心交易包含公对公服务费、私对私留利等复杂的资金分流。线下推进导致信息存在壁垒、财务核算困难、资料流转缺乏留痕。

### 1.2 产品目标

为“Xuwu Ledger”打造一个闭环的线上化业务流转与财务结算中枢，实现：

* **流程标准化：** 从提出需求到开票完结，全链路节点可追踪。
* **财务透明化：** 自动记录服务费、留利、提成，提供月度流水与纳税估算依据。
* **协同高效化：** 资料在线审核，状态变更实时通知，降低沟通成本。

## 2. 角色与权限定义 (RBAC)

系统需建立严格的数据隔离机制，各角色仅能操作和查看权限范围内的数据。

| 角色名称 | 核心职责与权限范围 | 数据可见性 |
| --- | --- | --- |
| **系统管理员 (Admin)** | 统筹全局，管理账号、权限、系统配置。 | 全局所有数据。 |
| **运营/审核 (Ops)** | 审核客户资料，确认链条方反馈，管理合同与发票，统筹全局资金账单。 | 全局业务数据、完整财务报表。 |
| **销售 (Sales)** | 建立客户档案，发起业务需求，督促客户提交资料，协助流转。 | **仅可见自己名下的客户**、跟进中的业务进度、**个人业绩与提成报表**。 |
| **客户 (Client)** | 接收业务需求，上传合规资料，查看本人关联的账单流水、合同及开票进度。 | **仅可见本企业/本人的业务进度与流水**。 |
| **链条方 (Partner)** | （可选登录）接收运营派发的资料，反馈买卖方（公司A/公司B）的合作确认信息。 | 仅可见被分配的业务订单及对应资料。 |

## 3. 核心功能模块设计

### 3.1 统一认证与工作台模块

* **多端登录：** 内部员工（销售、运营）通过账号密码登录；客户可通过手机号验证码或销售分配的初始账号登录。
* **角色工作台 (Dashboard)：**
* **销售端：** 展示“本月新增客户”、“待客户上传资料清单”、“我的本月预估提成”。
* **运营端：** 展示“待审核资料列表”、“待签合同预警”、“本月全局流水看板”。
* **客户端：** 展示“待办事项（如：需补充资料、需签合同）”、“我的历史订单”。



### 3.2 CRM 与客资管理模块

* **客户档案管理：** 销售录入客户基础信息（企业名称、联系人、资质等），系统自动建立“销售-客户”的强绑定关系，作为后续提成核算的唯一基准。
* **需求立项：** 销售针对特定客户发起“业务需求单”，标志着一个完整业务生命周期的开始。

### 3.3 资料流转与审批中心

这是打通销售、客户与运营的核心枢纽。

* **资料提交流水线：**
* 支持客户自主上传或由销售代客上传（营业执照、对账单等）。
* 支持文件预览和打包下载功能。


* **运营审核工作流：**
* 运营对提交的资料进行“通过”或“驳回”操作。
* 若驳回，需必填驳回原因，系统打回至销售/客户工作台。


* **链条方确认：** 运营审核通过后，将资料流转给链条方，链条方确认“可以做”后，业务进入资金流转阶段。

### 3.4 财务流水与结算中心 (核心)

负责将时序图中的复杂资金流进行数字化拆解。

* **订单账单拆解记录：** 每笔业务必须录入“总流水金额”，并强制拆分为两部分录入：
1. **服务费：** 记录转入公司对公账户的金额。
2. **留利：** 记录转给个人账户的金额。


* **实际打款记录：** 记录公司转给客户的实际货款（流水数据），以及个人转给链条方的留利费用，确保资金流向可查。
* **销售提成核算：** 根据绑定的客户订单，系统依据预设的提成规则自动生成销售个人的提成明细单。
* **月度财务报表与税务辅助：**
* 自动汇总当月对公总入账、对私总入账。
* 结合对公入账流水与后续的开票金额，输出“月度纳税基数估算表”，供财务参考。



### 3.5 合同与票税管理模块

* **合同节点跟进：**
* 记录运营与链条方签订合同的状态与归档。
* 记录客户跟公司签订合同的状态，支持提醒客户签约。


* **开票管理：**
* **进项记录：** 记录公司B开给本公司的发票（扣除等值留利额后的金额）。
* **销项记录：** 记录本公司开给客户的服务费发票。
* 系统自动核对发票金额与对公入账金额是否平账。



### 3.6 消息与通知机制

* **站内信与系统提醒：**
* **资料状态变更：** 运营审核完毕后，系统自动发送消息给对应销售：“您跟进的客户 [XX公司] 资料已审核通过，请推进下一步。”
* **待办催办：** 销售可一键触发“催办”指令，客户登录后会收到醒目的上传资料/签合同提醒。

## 4. 业务状态机 (业务全生命周期流转)

每个“业务单”在系统中将经历以下标准状态，状态的变更由特定角色的操作触发：

1. **需求新建** (销售操作)
2. **待传资料** (客户/销售操作)
3. **运营待审** (运营操作)
4. **链条方确认中** (链条方反馈)
5. **资金拆解录入** (销售/运营操作)
6. **合同签署中** (多方操作)
7. **流水拨付与开票** (财务/运营操作)
8. **订单完结**
